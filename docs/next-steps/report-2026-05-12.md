## Pending: Manual Secrets Required Before First Deploy

### Before Wave 15 — Database RGD

**`gen3-aurora-master-password`** must exist in AWS Secrets Manager before `database1-rg.yaml` runs. Aurora's ACK controller reads this secret to set the master DB password at cluster creation time.

```json
{ "username": "postgres", "password": "<STRONG_PASSWORD>", "host": "placeholder", "port": "5432" }
```

**`host` stays as a placeholder permanently — the RGD never updates this field.** The real Aurora endpoint is surfaced in the `databaseBridge` ConfigMap and injected into gen3-helm as `global.postgres.master.host` at ArgoCD sync time. gen3-helm's db-init jobs also read this secret (via `global.postgres.externalSecret`) for the `username` and `password` fields when creating per-service databases — but again, `host` is overridden by the separate helm parameter.

### Before Wave 35 — Fence secrets

Three secrets Fence always pulls from AWS SM. ExternalSecrets fail if any are missing.

#### A. Create Google OAuth2 credentials

1. Go to [Google Cloud Console → APIs & Services → Credentials](https://console.cloud.google.com/apis/credentials)
2. Click **Create Credentials → OAuth 2.0 Client ID**; Application type = **Web application**
3. Add Authorized redirect URI: `https://spoke1dev.rds-pla.net/user/login/google/login/`
4. Click **Create** — note the `client_id` and `client_secret`

#### B. Generate an encryption key

```bash
ENCRYPTION_KEY=$(python3 -c "import base64,os; print(base64.urlsafe_b64encode(os.urandom(32)).decode())")
echo "$ENCRYPTION_KEY"   # save this value
```

#### C. Create `fence-config`

The SM secret value must be a single key `fence-config.yaml` containing the YAML string below. Replace the three placeholders.

```bash
aws secretsmanager create-secret --name fence-config --region us-east-1 \
  --secret-string "$(cat <<'EOF'
{
  "fence-config.yaml": "APP_NAME: \"Gen3 Data Commons\"\nBASE_URL: \"https://spoke1dev.rds-pla.net/user\"\nENCRYPTION_KEY: \"<ENCRYPTION_KEY>\"\nOPENID_CONNECT:\n  google:\n    client_id: \"<GOOGLE_CLIENT_ID>\"\n    client_secret: \"<GOOGLE_CLIENT_SECRET>\"\n    discovery_url: \"https://accounts.google.com/.well-known/openid-configuration\"\n    redirect_url: \"https://spoke1dev.rds-pla.net/user/login/google/login/\"\n"
}
EOF
)"
```

#### D. Create Google creds placeholders

Required by ExternalSecret even if Google Cloud Storage is not in use.

```bash
aws secretsmanager create-secret --name fence-google-app-creds-secret --region us-east-1 \
  --secret-string '{"fence_google_app_creds_secret.json": "{}"}'

aws secretsmanager create-secret --name fence-google-storage-creds-secret --region us-east-1 \
  --secret-string '{"fence_google_storage_creds_secret.json": "{}"}'
```

## Pending: WTS OIDC — Bootstrap Dependency (Post-Fence)

WTS credentials are **issued by Fence** at client registration time. They cannot be pre-created — Fence must be healthy and reachable first.

#### Step 1 — Confirm Fence is healthy

```bash
kubectl get pods -n gen3 -l app=fence
# All pods should be Running/Ready before proceeding
```

#### Step 2 — Register WTS as an OIDC client

```bash
kubectl exec -n gen3 \
  $(kubectl get pod -n gen3 -l app=fence -o jsonpath='{.items[0].metadata.name}') \
  -- fence-create client-create \
    --client wts \
    --urls https://spoke1dev.rds-pla.net/wts/oauth2/authorize \
    --username wts_service_account \
    --type confidential \
    --grant-types authorization_code refresh_token client_credentials
```

Note the `client_id` and `client_secret` from the output.

#### Step 3 — Create `gen3-wts-oidc-client`

```bash
aws secretsmanager create-secret --name gen3-wts-oidc-client --region us-east-1 \
  --secret-string '{"client_id": "<CLIENT_ID>", "client_secret": "<CLIENT_SECRET>"}'
```

#### Step 4 — Create `gen3-wts-g3auto`

```bash
ENC=$(python3 -c "import base64,os; print(base64.urlsafe_b64encode(os.urandom(32)).decode())")
SK=$(python3 -c "import base64,os; print(base64.urlsafe_b64encode(os.urandom(32)).decode())")

aws secretsmanager create-secret --name gen3-wts-g3auto --region us-east-1 \
  --secret-string "$(cat <<EOF
{
  "appcreds.json": "{\"wts_base_url\": \"https://spoke1dev.rds-pla.net/wts/\", \"encryption_key\": \"$ENC\", \"secret_key\": \"$SK\", \"fence_base_url\": \"https://spoke1dev.rds-pla.net/user\", \"oidc_client_id\": \"<CLIENT_ID>\", \"oidc_client_secret\": \"<CLIENT_SECRET>\", \"external_oidc\": []}"
}
EOF
)"
```

## Can These Secrets Be Deployed as RGDs?

The intent is to eliminate CLI commands and wire everything as KRO graph dependencies instead.

| Secret | Via RGD? | Notes |
|--------|---------|-------|
| `gen3-aurora-master-password` | ✅ Yes | ACK `Secret` resource in a new pre-deploy RGD; password field can be a schema spec input |
| `fence-config` | ✅ Partial | RGD can create the SM secret using ACK `Secret`; the Google `client_id`/`client_secret` must still be obtained from Google Cloud Console first, then passed as RGD spec fields |
| `fence-google-app-creds-secret` | ✅ Yes | RGD can create the placeholder `{}` version immediately; no external dependency |
| `fence-google-storage-creds-secret` | ✅ Yes | Same as above |
| `fence-jwt-keys` | ✅ Already handled | fence PushSecret creates this automatically on first pod init |
| `gen3-wts-oidc-client` | ❌ No | Must come from `fence-create` on a running Fence pod — Fence is the issuing authority |
| `gen3-wts-g3auto` | ❌ No | Depends on the OIDC credentials from the step above |

### How to structure a pre-deploy secrets RGD

Create a new RGD (e.g. `awsgen3secrets1`) that runs before Wave 35, containing ACK `Secret` resources:

```yaml
- id: fenceGoogleAppCreds
  template:
    apiVersion: secretsmanager.services.k8s.aws/v1alpha1
    kind: Secret
    metadata:
      name: ${schema.spec.name}-fence-google-app-creds
      namespace: ${schema.spec.namespace}
      annotations:
        services.k8s.aws/region: ${schema.spec.region}
        services.k8s.aws/adoption-policy: adopt-or-create
        services.k8s.aws/deletion-policy: retain
    spec:
      name: fence-google-app-creds-secret
      stringData: '{"fence_google_app_creds_secret.json": "{}"}'
      tags:
        - key: ManagedBy
          value: Kro
```

The fence-config secret requires `ENCRYPTION_KEY` and Google credentials as schema spec inputs, giving you a fully graphed deployment where those two values are the only external prerequisites.

**WTS remains the only secrets that cannot be graphed** — they are fundamentally post-Fence because Fence is their issuing authority.

**How PushSecret works:**
1. db-init Job creates a Kubernetes Secret `{service}-dbcreds-bootstrap` with random credentials
2. PushSecret controller pushes the K8s Secret to AWS Secrets Manager under key `{externalSecrets.dbcreds}`
3. ExternalSecret then pulls from AWS SM back into the namespace for the running pod
