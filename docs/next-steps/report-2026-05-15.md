## Issue Tracker

### Issue 1 — `messaging` disabled, `audit-g3auto` will block forever

**Severity: High**

`audit-g3auto-secret1-rg.yaml` reads from `messaging-bridge` (externalRef
`messagingConfigMapName: "messaging-bridge"`). The `messaging` instance is `enabled: false`
in spoke1 values, so `messaging-bridge` never exists. When `auditG3AutoSecret` is enabled
its Job depends on `messagingConfig` — the RGD will stay NotReady until `messaging` is
turned on first.

**Fix:** Enable `messaging` before enabling `auditG3AutoSecret`.

---

### Issue 2 — `app-helm1` hard-depends on `appIam` and `platformHelm` (both disabled)

**Severity: High**

`app-helm1-rg.yaml` has strict `readyWhen` on:
- `appIamBridge` → `fence-role-arn`, `audit-role-arn`, `external-secrets-role-arn`, etc. must all be non-empty
- `platformHelmBridge` → `argocd-cluster-name` must be non-empty

Both source from disabled instances. Enabling `appHelm` without first enabling and
reconciling `appIam` and `platformHelm` means the RGD graph will never resolve.

**Fix:** Enable and wait for `appIam` + `platformHelm` before enabling `appHelm`.

---

### Issue 3 — `app-helm1` and all secret RGDs share wave 35 (no ordering)

**Severity: Medium**

`awsgen3apphelm1` and all Phase4 secret RGDs (`fenceConfigSecret`, `wtsG3AutoSecret`, etc.)
are annotated `argocd.argoproj.io/sync-wave: "35"`. ArgoCD does not guarantee order within
the same wave. If app-helm is enabled simultaneously with the secret RGDs, the Gen3
ArgoCD Application may be created before the SM secrets contain values — services will
start with empty ExternalSecret syncs and fail.

**Recommended fix:** Promote secret seeding RGDs to wave 34 and keep `app-helm1` at 35.
This enforces: secrets seeded → app deployed.

```yaml
# In each secret RGD:
annotations:
  argocd.argoproj.io/sync-wave: "34"
```

---

### Issue 4 — WTS OIDC chicken-and-egg

**Severity: Medium**

`wts-g3auto-secret1-rg.yaml` reads `wts-oidc-client-id` and `wts-oidc-client-secret`
from `gen3-secret-bootstrap`. These values come from running
`fence-create client-create` **after** Fence is deployed and running. They cannot
exist in the bootstrap secret at the time of initial app deploy.

**Recommended approach:** Do not enable `wtsG3AutoSecret` on the first deploy.
Deploy the core app first with `wts.enabled: true` in gen3-values (WTS will wait for
its ExternalSecrets but not crash). After Fence is healthy, run `fence-create`,
add the WTS OIDC keys to the bootstrap secret, then enable `wtsG3AutoSecret`.

---

### Issue 5 — `fence-config.yaml` in bootstrap is missing `ENCRYPTION_KEY`

**Severity: Medium**

`secrets.yaml` (the bootstrap template) still has:
```yaml
ENCRYPTION_KEY: "<base64-urlsafe-32-bytes>"
```
The `fence-config-secret1-rg.yaml` Job does a non-empty existence check only — if the
secret value string is non-empty it will not error, but Fence **will fail to start** when
it tries to decrypt tokens with a placeholder key.

**Generate before applying:**
```bash
python3 -c "import base64,os; print(base64.urlsafe_b64encode(os.urandom(32)).decode())"
```

---

### Issue 6 — `metadata-g3auto` bootstrap `dbcreds.json` has placeholder host

**Severity: Low**

`dbcreds.json` in the bootstrap template contains `"db_host": "<AURORA_ENDPOINT>"`.
This value is superseded by ESO PushSecret (which populates per-service DB creds from
Aurora after cluster is up), so metadata will ultimately use the real host. However,
if the bootstrap Job runs before ESO has pushed the real metadata creds, and something
reads `metadata-g3auto` before ESO syncs, it will see the placeholder.

**Recommendation:** Leave `dbcreds.json` in bootstrap as-is (the ESO push wins) but
document the timing dependency clearly in the bootstrap secret comments.

---

### Issue 7 — `manifestservice` bucket is empty, MANIFEST_BUCKET_NAME will be blank

**Severity: Low (service disabled)**

`manifestBucketName: ""` in spoke1 `infrastructure-values.yaml` means
`infrastructure-values` ConfigMap key `storage-manifest-bucket-name` will be empty.
`manifestservice-g3auto-secret1-rg.yaml` injects this as the `MANIFEST_BUCKET_NAME`
env var — if enabled while the bucket name is empty, the seeded secret will contain
`"manifest_bucket_name": ""`.

**Fix:** Either keep `manifestserviceG3AutoSecret: enabled: false` until the bucket is
provisioned (set `manifestBucketName` in storage values so Storage1 creates the bucket),
or leave `manifestservice.enabled: false` in gen3-values until Storage1 is configured.

---

### Issue 8 — `fenceGoogleAppCredsSecret` and `fenceConfigSecret` are disabled but gen3-values relies on them

**Severity: Medium**

`gen3-values.yaml` sets:
```yaml
fence:
  externalSecrets:
    createK8sFenceConfigSecret: false      # disables Helm native secret creation
    createK8sJwtKeysSecret: false
    createK8sGoogleAppSecrets: false
```
These flags tell gen3-helm **not** to create K8s Secrets for fence-config, JWT keys, and
Google app creds — it expects ExternalSecrets to source them from AWS SM. But
`fenceConfigSecret: enabled: false` and `fenceGoogleAppCredsSecret: enabled: false` mean
the SM secrets are never seeded. Fence will crash because its ExternalSecrets find empty
or non-existent SM values.

**Fix:** Enable and seed `fenceConfigSecret` and `fenceGoogleAppCredsSecret` **before**
enabling `appHelm`. If you prefer Helm to create the secrets natively, flip the
`createK8s*` flags back to `true` and the bootstrap RGDs are not needed.

---

### Issue 9 — `spokeAccess` / `platformHelm` disabled but required by `app-helm1`

**Severity: High** (same as Issue 2 — captured here for the record)

The ArgoCD Application created by `app-helm1` targets `${platformHelmBridge.data['argocd-cluster-name']}`.
Without `spokeAccess` creating the ArgoCD cluster Secret, the spoke cluster name will be
empty and the Application will be created targeting an unknown cluster.

## Recommended Deployment Sequence

### Phase A — Infrastructure (waves 15–27, enable one at a time, verify each)

```
1. Enable: networkSecurity, domainSecurity, messaging  (wave 15)
2. Enable: compute, storage                             (wave 20)
3. Enable: secretBootstrap                              (wave 26)
4. Enable: database                                     (wave 27)
5. Enable: spokeAccess, platformIam, appIam             (wave 25)
6. Enable: platformHelm                                 (wave 30)
```

### Phase B — Populate bootstrap secret

```bash
# After rotating the Google credential (Issue 1):
kubectl create secret generic gen3-secret-bootstrap -n spoke1 \
  --from-literal=fence-config.yaml="$(cat /path/to/fence-config.yaml)" \
  --from-literal=fence_google_app_creds_secret.json='{}' \
  --from-literal=fence_google_storage_creds_secret.json='{}' \
  --from-literal=base64Authz.txt="$(echo -n 'user:password' | base64)" \
  --from-literal=dbcreds.json='{"db_host":"placeholder","db_username":"metadata","db_password":"placeholder","db_database":"metadata"}' \
  --from-literal=metadata.env='GEN3_PUBLIC_DATASETS=false' \
  --from-literal=indexd-fence="$(openssl rand -base64 24)" \
  --from-literal=indexd-sheepdog="$(openssl rand -base64 24)" \
  --from-literal=indexd-ssj="$(openssl rand -base64 24)" \
  --from-literal=indexd-gateway="$(openssl rand -base64 24)"
  # wts keys: add AFTER fence-create client-create (see Issue 4)
```

### Phase C — Enable secret RGDs (wave 34, before app-helm)

Enable in spoke1 values, verify each Job succeeds:
```
fenceConfigSecret:          enabled: true   # requires fence-config.yaml filled in bootstrap
fenceGoogleAppCredsSecret:  enabled: true   # can be {} placeholder if not using Google Cloud
fenceGoogleStorageCredsSecret: enabled: true
auditG3AutoSecret:          enabled: true   # requires messaging enabled (Issue 1)
metadataG3AutoSecret:       enabled: true
indexdServiceCredsSecret:   enabled: true
# wtsG3AutoSecret:          leave false until fence-create completes
# manifestserviceG3AutoSecret: leave false until bucket is configured
```

### Phase D — Deploy core app (wave 35)

```
appHelm: enabled: true
```

Verify:
```bash
kubectl get awsgen3apphelm1 -n spoke1
kubectl get application gen3-app-helm -n argocd
kubectl get externalsecret -n spoke1
```

### Phase E — After Fence is healthy — enable WTS secret

```bash
# Run fence-create client-create to get WTS OIDC credentials
# Then patch the bootstrap secret with the new keys
kubectl patch secret gen3-secret-bootstrap -n spoke1 --type=merge \
  -p '{"stringData":{"wts-encryption-key":"<key>","wts-secret-key":"<key>","wts-oidc-client-id":"<id>","wts-oidc-client-secret":"<secret>"}}'
# Enable
# wtsG3AutoSecret: enabled: true
```
